### **Решение**
Выбрано технологическое решение на базе **фреймворка Spring Batch**. 

**Почему Spring Batch:**
1. Команда разработки уже исторически глубоко погружена в стек Java/WildFly. Переход на Spring Boot/Spring Batch для автономных ETL-ворикеров пройдет с минимальным кривой обучения (в отличие от внедрения Python Airflow).
2. Архитектура `Chunk Oriented Processing` (чтение FlatFile -> процессинг -> запись JdbcBatchItemWriter) идеально решает проблему "построчных вставок", заменяя тысячи долгих транзакций на несколько крупных блоков (например, по 500 строк).
3. Легко реализуется параллельная обработка (Partitions), когда один гигантский файл делится на куски, которые обрабатывают разные потоки.

**C4 Диаграмма Контекста (To-Be)**

### **Альтернативы**
Опишите здесь наиболее важные альтернативные решения.
1. **Apache Airflow:** Де-факто стандарт для ETL пайплайнов. Дал бы удобную визуализацию графов работы. Отвергнут, так как требует поднятия Python инфраструктуры и найма дата-инженеров; для простого обогащения 1 таблицы это оверкилл.
2. **K8s CronJob с Bash-SQL утилитой:** Использовалось в предыдущих задачах. Отлично для синхронизации, но здесь требуется сложная бизнес логика проверки DTO (запрос лояльности и трансформация в Java-коде), что на чистом SQL делать опасно и трудно поддерживать.

**Недостатки, ограничения, риски**
- **Риск:** Spring Batch-приложения могут потреблять значительный объем JVM Heap Memory, если неправильно настроить размер 'Chunk-ов' на объемах в ~1.5 млн строк.
- **Ограничения:** Возникнет необходимость поддержки отдельной схемы метаданных (Spring Batch Metadata Tables) в базе данных, чтобы оркестратор отслеживал упавшие джобы.
- **Недостатки:** Пока компонент не вынесен в микросервис, он все еще будет разделять процессорное время сервера с монолитом (если запущен на той же VM Compute Engine). Возможна просадка производительности, рекомендуется вынесение Spring Batch в отдельный Docker контейнер / K8s Pod на независимых ресурсах.
