# Задание 2. Разработка дизайна модуля пакетной обработки данных

## 1. Обоснование выбора решения

Для задачи ежедневной выгрузки прайс-листов B2B-клиентам в 06:00 утра (объем данных — сотни килобайт: тысячи товаров и сотни клиентов) необходимо было выбрать архитектурный подход: Spring Batch, Apache Airflow, K8s Job, или Spark.

Основываясь на анализе вводных, было выбрано решение **K8s Job (Kubernetes CronJob)**.

### Сравнительная таблица

| Критерий                                                      | Spring Batch                                                                | Apache Airflow                                                       | K8s Job (CronJob)                                                                                                     | Apache Spark                                                             |
|:--------------------------------------------------------------|:----------------------------------------------------------------------------|:---------------------------------------------------------------------|:----------------------------------------------------------------------------------------------------------------------|:-------------------------------------------------------------------------|
| **Наличие конфигурации CRON-расписания**                      | Нет (требует обвязки в виде дополнительного шедулера или внешнего триггера) | Да (содержит мощный встроенный оркестратор)                          | **Да** (нативная абстракция `CronJob` в Kubernetes)                                                                   | Нет (не имеет своего шедулера, требует запуска извне)                    |
| **Сложность реализации логики по обработке данных**           | Средняя (много boilerplate-кода на Java для простого SQL to CSV вызова)     | Низкая (написание простого Python DAG)                               | **Очень низкая** (скрипт или легкая утилита с одним SQL-запросом)                                                     | Высокая (неоправданно сложная инициализация Spark, сборок и кластера)    |
| **Ресурсоёмкость (количество потребляемых ресурсов)**         | Средняя (постоянно работающий сервис или JVM, прогрев и аллокация памяти)   | Высокая (постоянно запущенные веб-сервер, шедулер и база метаданных) | **Минимальная** (ресурсы потребляются только несколько секунд в момент работы, затем полностью возвращаются кластеру) | Максимальная (огромный оверхед на поддержание кластерной инфраструктуры) |
| **Масштабируемость под нагрузкой и сложность реализации**     | Идеальная поддержка chunk обработки, но реализация партиционирования сложна | Горизонтально масштабируется воркерами, но сложная настройка HPA     | **Отлично масштабируется** (увеличивается через limits/requests или concurrent jobs из коробки)                       | Избыточная (предназначена для обработки терабайтов, а не мегабайта)      |
| **Сложность развёртывания в облаке и интеграция**             | Низкая (обычный Docker контейнер)                                           | Высокая (поднятие и поддержка Helm-чарта Airflow)                    | **Нулевая** (БД уже в k8s, нужно просто применить 1 `.yaml` манифест)                                                 | Крайне высокая (поднятие Spark Operator или managed кластера)            |
| **Удобство интеграции с системами логирования и мониторинга** | Отлично (Spring Boot Actuator, ELK)                                         | Хорошо (Метрики для StatsD/Prometheus, UI)                           | **Отлично** (Stdout логи пода автоматически собираются демонами логгирования k8s и кладутся в ELK/Grafana)            | Средне (собирается через Spark History Server)                           |

### Выводы
Запускать тяжелые кластеры (Airflow или Spark) ради агрегации 10 000 строк, которая представляет собой моментальную операцию SQL `JOIN` внутри БД — это грубый архитектурный паттерн (over-engineering), влекущий бессмысленные расходы на облачную инфраструктуру.
Так как микросервисы (и сама база) уже развернуты в Kubernetes, нативным и самым эффективным решением будет **Kubernetes CronJob**. Поднимется крошечный контейнер, выполнит запрос, выгрузит CSV куда требуется, и умрет, стопроцентно освободив ресурсы. Это дешево, быстро и надежно.

## 3. Набросок верхнеуровневого плана имплементации и конфигурации

Чтобы перенести дизайн в рабочий код, команде потребуется выполнить следующие 4 шага:

### Шаг 1. Разработка сервиса выгрузки (Утилиты)
*   **Действие:** Написать легковесный скрипт (Golang, Python или просто Bash-сценарий), который будет подключаться к базе.
*   **Имплементация:** Внутри скрипта выполняется запрос агрегации. Так как база это PostgreSQL, проще и быстрее всего использовать команду `COPY`:
    ```sql
    COPY (
      SELECT p.name, c.name, cp.price 
      FROM products p 
      JOIN client_prices cp ON ...
    ) TO STDOUT WITH CSV HEADER;
    ```
*   **Завершающее действие:** Скрипт сразу утилизирует полученный буфер в AWS S3 (используя SDK или AWS CLI) и завершается с кодом 0 (success). Выделять память на хранение файла на диске даже не понадобится.

### Шаг 2. Контейнеризация и публикация (Docker)
*   упаковать скрипт в минималистичный Docker Image (например, `alpine` или `distroless` образ), чтобы он потреблял не более 10-20 мегабайт ОЗУ.
*   Отправить образ в локальный контейнер-реестр (Docker Registry).

### Шаг 3. Подготовка манифеста K8s CronJob
Написать `yaml` конфигурацию, описывающую инфраструктурные аспекты решения. Ключевые параметры:
*   `schedule: "0 6 * * *"` — запуск строго каждый день в 06:00 по расписанию кластера.
*   `restartPolicy: OnFailure` — в случае моргания сети, k8s попытается перезапустить задачу.
*   `resources.limits: {memory: "128Mi", cpu: "100m"}` — ограничить ресурсы, чтобы job не отъел их у критического продакшена.
*   Подмонтировать `Secret` или настроить External Secrets K8s, чтобы прокинуть логин/пароль от базы и ключи от S3 как Environment Variables безопасно, не зашивая их в код.

### Шаг 4. Деплой и Мониторинг
*   **Деплой:** Применить конфигурацию к кластеру командой `kubectl apply -f cronjob.yaml` (или упаковать в существующий Helm chart интернет-магазина).
*   **Мониторинг:** Так как это стандартный объект K8s, любой упавший с ошибкой CronJob будет мгновенно замечен инфраструктурными метриками. Утилиты вроде `kube-state-metrics` автоматически отправят Alert в Slack/Telegram разработчикам, если расписание было сорвано, а логи выполнения будут в безопасности сохранены с помощью Promtail/Fluentd, собирающего `stdout` из скрипта-выгрузчика.
