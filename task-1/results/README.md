# Задание 1. Выбор и реализация решения для пакетной обработки данных

## Обоснование выбора: Apache Airflow

В качестве решения для организации пакетной обработки данных выбран **Apache Airflow**. Данный инструмент является индустриальным стандартом для оркестрации DAG обработки данных и наилучшим образом удовлетворяет всем выдвинутым требованиям.

### 1. Интеграция с внешними системами
Airflow обладает огромным списком готовых провайдеров (`apache-airflow-providers`), которые позволяют интегрироваться с практически любой базой данных или платформой без написания низкоуровневого кода взаимодействия по API:
*   **BigQuery и Redshift:** Существуют готовые операторы (`BigQueryInsertJobOperator`, `RedshiftSQLOperator` и другие), которые берут на себя авторизацию по ключам (через Airflow Connections) и выполнение запросов, ускоряя разработку в несколько раз. 
*   **Spark:** `SparkSubmitOperator` может отдавать задачи Spark-кластеру и дожидаться их завершения.
*   **Kafka:** Через провайдер `apache-airflow-providers-apache-kafka` возможна как отправка, так и чтение сообщений (ConsumeFromTopicOperator).

### 2. Поддержка ветвлений, условных операторов и event-triggers
*   **Ветвление (Branching):** Поддерживается "из коробки" с помощью операторов ветвления (например, `BranchPythonOperator`). Логика ветки определяется результатом возвращаемого значения Python-функции. 
*   **Event-triggers (Датчики событий):** Сенсоры (Sensors) в Airflow позволяют приостанавливать выполнение задачи, пока не наступит определенное событие: например, появление файла в S3/на диске (`FileSensor`), либо запись в БД.

### 3. Наличие out-of-the-box fallback-logic, retry и email-уведомлений
В Airflow всё это встроено на уровне конфигурации задачи (или всего пайплайна сразу в `default_args`):
*   **Повторные попытки (retry):** Параметры `retries=3` и `retry_delay=timedelta(minutes=5)` дадут команду Airflow автоматически повторить падающий кусок логики трижды с интервалом в 5 минут, прежде чем задача будет окончательно помечена как `failed` (демонстрируется в данном POC).
*   **Fallback логика:** При помощи параметра `trigger_rule` (например, `all_failed` или `one_failed`) можно реализовать "спасительные" (fallback) задачи, которые запустятся, только если основная ветка упадет.
*   **Email-уведомления:** При настроенном SMTP-сервере включение отправки письма при ошибке сводится к одному параметру: `email_on_failure=True`.

### 4. Развертывание в облачной среде (Cloud Native)
Airflow идеально подходит для облака. Он масштабируется (благодаря Celery или Kubernetes executor'ам) для обработки миллионов записей (сами данные он не перекладывает, он запускает тяжелые воркеры в БД/BigQuery/Spark). 
Его можно развернуть:
*   В **Kubernetes cluster** при помощи официального Helm Chart (один из самых популярных способов на проде).
*   Воспользоваться managed-решениями от облачных провайдеров: **Cloud Composer (GCP), Amazon MWAA, Yandex Managed Service for Apache Airflow**. Это полностью снимает с инфраструктурной команды бремя поддержки кластера.

## 2. Развертывание Proof of Concept (POC)

Папка `poc` содержит локальную демонстрацию упрощенного пайплайна Airflow.

### Описание графа (`batch_processing_dag.py`)
Пайплайн включает в себя необходимые шаги:
1.  **Чтение из источника данных** (`read_data_source`): Имитирует извлечение данных и вычисляет качество данных.
2.  **Анализ данных и ветвление** (`analyze_and_branch`): Если качество высокое, идем по успешной ветке. Если низкое (сбой форматирования и т.д.) - по ветке с ошибкой.
3.  **Успешное завершение + Уведомление**: При успешной обработке отправляется email-уведомление.
4.  **Установка retry-политики и Fallback**: Ветка ошибки (низкое качество) специально падает, запуская механизм retry (2 попытки с паузой 10 секунд). Если все попытки исчерпаны, DAG падает, и Airflow автоматически (из коробки `email_on_failure`) рассылает email о падении.
