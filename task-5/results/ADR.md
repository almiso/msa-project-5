### **Решение**

#### Сбор метрик: Micrometer → Prometheus (Pull-модель)

**Выбранный способ: Spring Boot Actuator + `micrometer-registry-prometheus`**

Spring Boot Actuator автоматически регистрирует все JVM/Spring Batch метрики через Micrometer. Endpoint `/actuator/prometheus` экспортирует их в текстовом формате Prometheus без единой строки кода. Prometheus сам приходит за метриками каждые 5 секунд (pull-модель).

**Почему Pull, а не Push (JMX Exporter, Pushgateway)?**
- Push-модель требует настройки агента (JMX) или смены архитектуры приложения.
- Pull-модель: Prometheus сам управляет частотой сбора и retry-логикой.
- Actuator уже в classpath Spring Boot — нулевые затраты на внедрение.

**Почему не JMX Exporter?** JMX работает по pull через Java-агент в JVM — требует `-javaagent` в команде запуска. Излишне сложно, когда Actuator решает ту же задачу проще.

#### Выбранные метрики и обоснование

| Метрика | PromQL | Обоснование |
|---------|--------|-------------|
| CPU usage | `process_cpu_usage * 100` | Основной сигнал перегрузки — исходная проблема TradeWare |
| JVM Heap used | `sum(jvm_memory_used_bytes{area="heap"})` | Утечки памяти при батч-обработке больших CSV — критический риск |
| Batch Items Written | `spring_batch_chunk_write_seconds_count` | Прямой индикатор прогресса ETL Job |
| Batch Job Runs | `spring_batch_job_seconds_count{name="importProductJob"}` | Подтверждение что Job вообще выполнялся |

#### Конфигурация оповещений

**Alert 1: High CPU Usage**
- Условие: `process_cpu_usage{job="batch-processing"} * 100 > 80`
- Период: срабатывает если условие держится > 1 минуты
- Реализация: Grafana Legacy Alert в JSON dashboard (работает без Alert Manager)

Выбрана Grafana (а не Prometheus alerting rules) — это проще для демонстрации, конфигурация хранится в provisioning JSON вместе с дашбордом.

#### Сбор логов: Filebeat → Logstash → Elasticsearch (Structured Logging)

**Выбранный способ: JSON stdout → Filebeat (docker input) → Logstash → Elasticsearch**

**Почему JSON-логи через stdout, а не запись в файл?**
- Docker-контейнеры по умолчанию захватывают stdout — никакого volume-mounting для лог-файлов.
- `logstash-logback-encoder` форматирует каждую строку лога в JSON с полями `level`, `msg`, `trace`, `class`, `thread` — Kibana может фильтровать по любому полю.
- Filebeat c типом `docker` автоматически добавляет метаданные контейнера (имя, образ, labels).

**Почему Filebeat, а не Logstash напрямую?**
- Filebeat — lightweight агент (< 20MB), не потребляет JVM heap.
- Logstash тяжелее (JVM), его роль — трансформация и маршрутизация. Разделение ответственности по принципу Unix.

**Почему не Fluentd?** Команда уже зафиксировалась на ELK. Смешивание с Fluentd создаёт ненужную сложность без выгоды для данного масштаба.

### **Альтернативы**

1. **Grafana Loki вместо ELK:** Loki не индексирует содержимое логов (только labels) — дешевле по хранилищу, но хуже для поиска по тексту сообщений. ELK выбран для полноценного full-text search.
2. **Prometheus Alertmanager:** Более мощный (routing, inhibition, silences). Отклонён как избыточный для 1-2 правил — Grafana alerts достаточно для текущего масштаба.

---

### **Недостатки, ограничения, риски**

- **Риск:** ELK стек потребляет ~1.5-2GB RAM. На машинах с < 8GB памяти возможна нестабильность Elasticsearch.
- **Ограничения:** Grafana Legacy Alerts (в JSON provisioning) устарели в Grafana 10+. При переходе на новый алертинг потребуется миграция на Unified Alerting + Contact Points.
- **Недостатки:** Filebeat фильтрует инфраструктурные контейнеры (logstash, elasticsearch, prometheus и т.д.) через `drop_event`. В проде рекомендуется использовать Docker labels для более гибкой фильтрации.
